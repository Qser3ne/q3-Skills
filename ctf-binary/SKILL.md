# CTF Binary

把当前任务视为一道正在比赛中的 Pwn 或 Reverse CTF 题目来处理，优先做本地分析、调试、复现和验证。

仅在“正在解 Pwn / Reverse CTF 题”时使用本 skill，即使用户没有显式写出 `$ctf-binary` 也可以触发；普通提到 CTF、泛泛讨论二进制安全时不应触发。

如果题目属于 Web 或 Misc，改用 `$ctf`；本 skill 不覆盖 Web、Misc、Crypto、Forensics。

## 核心约束

- 这是刚举办的比赛，网上不存在 WP。
- 优先分析题目附件、二进制、源码、IDA 数据库、核心转储、运行时报错和调试结果。
- 先做可验证的本地推理和实验，再决定下一轮利用或还原方向。
- 不要跳过静态分析和动态调试，避免只凭题名或函数名猜结论。

## 工具优先级

- 优先使用 `ida-mcp` 辅助查看函数、字符串、交叉引用、伪代码和关键控制流。
- 优先使用 `gdb` 做断点、单步、寄存器、栈、堆、内存布局和崩溃现场分析。
- 需要写利用脚本、交互脚本或本地/远程调试脚本时，优先使用 `pwntools`。
- 常规辅助工具如 `file`、`checksec`、`strings`、`objdump`、`readelf`、`ldd` 可直接配合使用。

## Reverse 方向约定

- 先确认架构、入口、关键函数、字符串、加解密逻辑、校验逻辑和输入输出路径。
- 对反编译结果中的关键判断、循环、位运算、表驱动和混淆逻辑，优先写小脚本做验证。
- 如果样本存在自修改、反调试、壳或运行时解密，优先结合 `ida-mcp` 和 `gdb` 拿真实行为，不要只看静态表象。

## Pwn 方向约定

- 先确认架构、保护项、链接库、交互协议、输入点、输出点和崩溃条件。
- 优先判断漏洞类型、利用原语、信息泄露条件、栈堆布局和远程本地差异。
- 在没有证据前，不要直接假定题目一定是某一种经典利用模板；先用调试结果验证偏移、地址和约束。

## Python 与 pwntools 约定

- 所有脚本默认使用 Python 3 执行。
- pwntools 虚拟环境根目录从 `SKILL-CTF-BINARY-PWNTOOLS-VENV` 环境变量读取。
- 因为变量名包含连字符，读取时使用 `printenv`，不要使用 shell 参数展开：

```bash
pwntools_venv="$(printenv 'SKILL-CTF-BINARY-PWNTOOLS-VENV' || true)"
```

- 需要导入 `pwntools` 的脚本，显式使用 `$pwntools_venv/bin/python`。
- 安装或补充 `pwntools` 相关依赖时，显式使用 `$pwntools_venv/bin/pip`。
- 不要把 `pwntools` 装到系统 Python，也不要默认装到工作区 `.venv`。
- 如果 `SKILL-CTF-BINARY-PWNTOOLS-VENV` 未设置或路径不可用，先要求用户设置该环境变量，不要猜测本机路径。

## 工具与依赖

- 如果要下载工具，尽量下载到当前工作区，例如 `./tools`、`./bin`、`./downloads`、`./tmp`。
- 优先使用工作区内的附件、脚本、核心文件、便携工具和容器，不要污染系统环境。
- 新建的辅助脚本、PoC、解题脚本也尽量放在当前工作区，并保持路径清晰。

## 本地与远程

如果有本地附件、二进制或 libc，优先完成本地静态分析和动态调试，再决定是否远程交互测试。如果题目只给远程服务，也要先尽量通过交互脚本收集可验证行为。

## 推荐工作流

1. 先盘点题目文件、架构、保护项、交互方式和可执行入口。
2. 用 `ida-mcp` 或常规静态工具先拿到函数结构、关键字符串、关键逻辑和潜在攻击面。
3. 用 `gdb` 复现关键路径、崩溃点、寄存器状态、栈堆布局或解密后的真实逻辑。
4. 用 `$pwntools_venv/bin/python` 编写最小验证脚本、调试脚本或 exploit/solve 脚本。
5. 再基于现有证据决定下一轮验证，而不是跳步。
