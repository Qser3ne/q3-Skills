# CTF

把当前任务视为一道正在比赛中的 Web 或 Misc CTF 题目来处理，优先做本地分析、复现和验证。

仅在“正在解 Web / Misc CTF 题”时使用本 skill，即使用户没有显式写出 `$ctf` 也可以触发；普通提到 CTF、泛泛讨论 CTF 时不应触发。

如果题目属于 Pwn 或 Reverse，改用 `$ctf-binary`；本 skill 不覆盖二进制方向。

## 核心约束

- 这是刚举办的比赛，网上不存在 WP。
- 优先分析题目附件、本地服务、源码、流量、协议和运行时报错。
- 先做可验证的本地推理和实验或者远程连接测试。
- 如果是 Web 方向的题目，不要在我提供的附件中寻找 flag。
- 如果出现超时、连接失败、间歇性 `500/502/503/504`、响应为空、行为突然与前序验证不一致等异常现象，先考虑靶机下线、服务崩溃、实例被回收、代理或本地网络访问异常等环境因素，不要直接把异常归因为漏洞特性或利用结果。
- 如果请求返回 `500`，先考虑靶机或服务可能已经崩溃、挂起或进入异常状态；必要时先询问用户是否需要重启靶机或确认当前服务状态。
- 如果后续怀疑靶机下线或网络访问异常，必须回看之前依据异常响应、超时、报错差异、状态码变化等现象得出的阶段性结论，分析这些结论是否可能被环境异常污染；未重新验证前，不要把它们当作稳定证据。



## Web 方向题目约定

- 除非题目明确指出存在 bot 或者题目有十分明确的 XSS 触发条件，否则不要以 XSS 为解题方向，因为即使存在 XSS 漏洞也无法利用。不得猜测题目隐藏了 bot 功能。

## Misc 方向题目约定

- 优先从附件格式、编码、压缩、图片音频隐写、流量、日志和脚本行为中寻找可验证线索。
- 对于批量解码、提取、转换、爆破和格式修复，优先写短小可复现的脚本，而不是手工重复操作。

## Python 约定

- 所有脚本默认使用系统 `python3` 执行。
- 不创建、不使用工作区隔离 Python 环境。
- 如果缺少所需 Python 库，直接使用 `python3 -m pip install <包名>` 下载并安装。
- 如果在 Linux 环境下安装依赖时出现系统环境保护相关警告或阻止安装，则追加 `--break-system-packages` 参数。

## 工具与依赖

- 如果要下载工具，尽量下载到当前工作区，例如 `./tools`、`./bin`、`./downloads`、`./tmp`。
- 优先使用工作区内的便携二进制、源码目录或容器。
- 新建的辅助脚本、PoC、解题脚本也尽量放在当前工作区，并保持路径清晰。

## 本地与远程

如果有必要在本地验证或测试题目行为，优先在本地复现环境并测试。如果没有类似需求，优先远程连接测试。若远程行为突然异常，先复测基础连通性、响应稳定性和可重复性；如果确认靶机或网络状态异常，应暂停基于该轮异常现象继续推理。

## Docker 的使用

- 如果题目提供了 docker 文件，本机有 Docker 环境，优先在本地起 Docker 或 Docker Compose 来复现题目服务。
- 先检查 `Dockerfile`、`docker-compose.yml`、启动脚本、环境变量、端口和挂载目录。
- 优先通过本地启动后的报错、日志、崩溃信息和行为差异推进分析，而不是先做大段猜测。
- 如果题目附件没有提供 Dokcer 容器，或者 Docker 服务不可用，再询问用户应该如何处理。



## 推荐工作流

1. 先盘点题目文件、服务类型、交互方式和可执行入口。
2. 如果有必要在本地验证或测试题目行为，优先在本地复现服务或脚本行为，优先拿到真实报错和日志。如果没有类似需求，优先远程连接测试。
3. 用系统 `python` 写小型验证脚本、数据处理脚本或 exploit/solve 脚本；缺少依赖时按 Python 约定直接安装。
4. 如果远程环境出现异常现象，先确认是不是靶机下线、实例回收或网络访问问题，再决定是否继续做漏洞推理。
5. 再基于现有证据决定下一轮验证，而不是跳步。
