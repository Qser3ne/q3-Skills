# CTF 中文解题报告

仅在用户显式调用 `ctf-wp` 时使用本 skill。不要因为用户泛泛提到“writeup”“解题报告”就自动套用。

目标：

- 产出详细中文 CTF 解题报告。
- 站在解题选手视角，按真实思考顺序写出观察、怀疑、验证、转向和拿到 flag 的过程。
- 保证读者可以依据报告中的关键证据、请求、命令和复现材料重走整条解题路径。

## 使用边界

- 仅在用户明确要求生成中文 CTF WP、解题报告或 writeup 时触发。
- 默认输出审校完成、可直接交付的最终版本，而不是只给提纲。
- 不要把报告写成赛后摘要、提交记录或纯结果汇总。

## 输入参数

- `challenge-name` 用于默认产物命名。
- 如果用户未传入 `challenge-name`，优先取题目目录名，其次取服务名或附件主名。
- 如果仍无法稳定推断，统一回退为 `unknown-challenge`。
- 默认文件名因此回退为 `./solve/writeup-unknown-challenge.md`。

## 产物规范

- 最终 WP 默认输出到 `./solve/writeup-<challenge-name>.md`。
- 最终复现入口默认输出到 `./solve/solve.<ext>`；语言以实际解题使用的语言为准。
- 如果题目天然需要多文件工程、交互式步骤或手工辅助，提供统一入口，如 `./solve/run.sh`、`./solve/Makefile` 或 `./solve/README.md` 中唯一的启动命令。
- 为最终复现直接服务的附件放入 `./solve/`。
- 草稿、一次性脚本、临时分析文件和中间验证材料可以留在 `solve/` 之外。
- 如果工作目录下不存在 `solve` 文件夹，应先创建再写入。

## 强制要求

- 必须是中文。
- 必须从选手视角按时间顺序组织，而不是倒叙公布答案。
- 必须包含 `错误尝试`、`转向理由`、`出错点复盘`、`耗时点复盘` 四类内容。
- 必须包含足以复现的关键请求、关键命令或关键交互。
- 必须给出完整复现入口，不能只贴零散代码片段。
- 必须解释关键判断依据，不能只写“测试发现”“分析可得”。
- 文中所有最终产物路径统一按 `./solve/` 目录描述。

## 写作原则

### 1. 顺序必须贴近真实解题过程

不要一上来直接公布最终解法。按选手的自然顺序写：

1. 题目初看时拿到了哪些信息
2. 最先想到的几个方向分别是什么
3. 为什么先验证其中某个方向
4. 验证结果说明了什么
5. 哪个现象让你意识到前一个方向不对
6. 你为何转向新方向
7. 新方向如何逐步收敛到最终利用链

### 2. 每一步都要回答“为什么”

每个关键动作都要说明理由，不要只记流水账。例如：

- 为什么怀疑这里存在注入、反序列化、越权、格式错误、弱随机、侧信道或逻辑漏洞
- 为什么某条报错、状态码、回显、二进制行为、日志痕迹或中间结果值得继续深挖
- 为什么某个 payload、脚本、爆破范围、偏移计算或解密方式是合理选择

### 3. 不要省略错误尝试

错误尝试是报告的一部分，必须保留，但不要编造。优先记录：

- 最初的错误假设
- 为验证错误假设做了什么
- 得到了什么结果
- 这些结果为什么足以排除该方向
- 排除后是如何把注意力转到新的突破口上的

只有在确实缺失材料时，才允许写“没有保留完整记录”。核心利用链、最终复现路径和最终结果不得依赖这句兜底。

### 4. 复现优先于文学性

报告应服务于复现。关键 payload、关键请求、关键命令、关键输出和复现入口必须给出，必要时补充环境说明、依赖版本、目录结构、目标地址、端口、题目附件路径、flag 格式和运行方式。

## 生成流程

### Step 1: 收集材料

从当前会话、题目目录和已有脚本中提取：

- 题目名、赛事名、类型、分值、flag 格式
- 附件、服务地址、账号、交互方式
- 解题过程中实际使用过的脚本、命令、payload、请求、响应片段
- 关键中间结果
- 错误尝试与转向线索
- 能支撑“出错点复盘”和“耗时点复盘”的证据，例如命令历史、失败输出、重复验证记录和长时间卡住的分支线索

可优先使用下面的方式快速扫描：

```bash
rg --files . | rg '(solve|exp|exploit|writeup|payload|script|wp|md|py|sh|js|java|go)$'
rg -n '(flag\\{|CTF\\{|ctf\\{|pwn|sql|ssti|xss|deserialize|jwt|pickle|offset|gdb|one_gadget|ROP)' .
```

如果是 Web 题，优先补齐：

- 关键 HTTP 请求样例
- 关键响应特征
- cookie、header、参数、编码方式
- 从哪一个响应细节看出突破点

如果是 Pwn / Crypto / Reverse / Misc 题，优先补齐：

- 交互协议或输入输出格式
- 偏移、地址、约束、数学关系或逆向定位依据
- 中间推导结果
- 最终复现入口

### Step 2: 先整理“思路链”，再落笔

先在脑中整理出下面这条链，再写正文：

`初始观察 -> 第一假设 -> 验证 -> 失败/不充分 -> 新线索 -> 新假设 -> 关键突破 -> 完整复现 -> flag`

如果某一步缺证据，不要硬写。应回头查脚本、命令历史、日志或输出。

### Step 3: 输出详细中文报告

正文必须以中文撰写，并优先采用下面的章节顺序。文中所有最终产物路径统一按 `./solve/` 目录描述。

## 推荐结构

```markdown
# <题目名>

## 题目信息

- 比赛：
- 类型：
- 分值：
- 题目附件 / 服务：
- flag 格式：

## 初始观察

从选手第一次接触题目开始写。说明你先看到了什么、为什么先关注这些信息、最先冒出的几个怀疑方向是什么。

## 第一轮尝试与碰壁

逐条写出最先做的验证。

### 尝试 1：<方向>

- 当时为什么会想到这个方向
- 具体做了什么
- 关键请求 / 命令 / 输入样例
- 结果是什么
- 为什么这个结果说明方向不对，或者至少不是最短路径
- 何时决定放弃该方向，以及转向依据是什么

### 尝试 2：<方向>

同上。可以有多轮，但不要堆无意义流水账，只保留真正影响思路转向的尝试。

## 关键转折

说明是哪一个现象、报错、输出、流量特征、反汇编位置、加密性质或脚本结果让你意识到真正突破口在哪里。这里必须明确写出“为什么这个线索重要”。

## 正确解法

按可复现顺序写完整解题过程。每一步都要解释理由。

### Step 1：<动作>

说明动作、理由和预期。

```bash
# 关键命令 / 请求 / 交互
```

必要时补关键输出：

```text
<关键输出片段>
```

### Step 2：<动作>

继续直到能够稳定拿到 flag。

## 关键请求与关键复现入口

### 关键请求样例

对 Web 题，优先给出精简但完整的 `curl` 或原始 HTTP 请求；对其他题型，给出关键命令或交互样例。参数、header、编码方式、特殊字段要保留完整。

### 最终复现入口

给出可以直接复现的完整入口，而不是几个零散片段。若可由单一脚本完成，入口默认放在 `./solve/solve.<ext>`；若需要多文件工程或手工配合，给出统一入口与逐步执行顺序。

```text
# ./solve/solve.<ext> 或其他统一入口
```

说明运行方式、依赖和预期输出；命令示例必须使用实际可执行命令，而不是固定写成某一种语言的示例。

## 复现说明

写清楚复现时需要准备什么、先运行什么、再观察什么、成功标志是什么。最终 WP、最终复现入口和相关附属文件都按 `solve/` 目录组织描述。如果题目存在不稳定因素，要写明如何重试或校准。

## 出错点复盘

单独复盘本次解题中真正拖慢进度或导致误判的关键错误。至少说明：

- 哪些判断、操作或方向后来证明是错误的
- 为什么这些地方当时容易误判
- 后来靠什么证据意识到这里错了

## 耗时点复盘

单独复盘本次解题中明显花费大量时间的排查环节。至少说明：

- 哪些环节反复排查、试错或验证
- 这些时间为什么会花得多
- 如果重做一次，哪些证据可以更早帮助排除错误方向或缩短排查时间

## 最终结果

给出 flag 或最终敏感结果。如果用户要求打码，再按要求处理。

## 总结

用几句话总结真正的突破点、最容易误判的地方、最耗时的环节，以及为什么前面的错误尝试看似合理但实际上走不通。
```

## 风格要求

- 允许详细，但不要故作抒情。
- 保持技术判断清晰，少写空话。
- 对关键证据和关键结论之间的因果关系写明白。
- 对截图依赖要谨慎。优先用可复制的文本、请求、命令和脚本表达。

## 禁止事项

- 不要把报告写成赛事提交摘要。
- 不要省略失败尝试。
- 不要把最终复现入口拆成多个无法独立落地的小段。
- 不要伪造未发生过的操作或结论。
- 不要只给 payload，不解释为什么是它。
