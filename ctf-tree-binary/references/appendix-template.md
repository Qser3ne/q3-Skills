# 附录模板

每个需要可复现证据、详细推理、代码、payload 或题目相关失败分析的任务，都使用一个独立附录文件。

附录文件名绑定任务序号，例如 `ctf-worklog/appendices/task-3.md`。

若阻塞原因只是 Docker Hub、远程靶机或其他外部基础设施连通性异常，则不要创建或更新本附录；应先在对话中告知用户。

## 骨架

```md
# 任务 3 附录

## 任务标识
- 任务序号：3
- 当前状态：待执行
- 前置结论：结论 2 [已确认]
- 产出结论：无

## 目标
验证当前任务目标是否成立，并为主树提供可复现、可续接的证据。

## 前置条件
- 二进制 / 样本路径：
- 架构 / 位数：
- libc / loader / 依赖文件：
- 远程地址 / 端口：
- 调试器 / 工具：
- 需要的脚本、输入样本、环境变量、容器：

## 可复现步骤
1. 执行命令或进入调试流程。

```bash
file ./chall
checksec --file=./chall
gdb -q ./chall
```

```gdb
b *main
run
info registers
x/16gx $rsp
```

```python
from pwn import *

io = process("./chall")
io.sendline(cyclic(200))
io.wait()
```

2. 如果存在第二步、第三步，继续按顺序写出，可混合记录命令行、`gdb`、`pwntools` 交互、反编译观察或样本处理过程。
3. 如果步骤依赖特定时序、重试次数、ASLR 状态、远程环境或并发条件，明确写出。

## 观测结果
记录关键输出、报错、寄存器状态、崩溃地址、偏移、内存片段、反编译差异或其他决定性片段。

```text
RIP 0x6161617461616173
offset = 72
```

## 推理过程
- 说明这些观测如何支持或否定当前任务目标。
- 如果当前任务仍然过宽，说明应如何收窄这个任务，而不是提前新增依赖它的后续任务。
- 如果排除了某个伪线索，也在这里说明原因。

## 产出结论
- 无

## 结果
- 任务状态更新：保持待执行 / 更新为已完成 / 更新为已失败
- 主文档同步动作：更新任务 3 在主树中的状态、证据与产出结论；如有新增或修正结论，同步更新对应结论块
- 主文档已同步：是 / 否；如果为否，写明未同步原因和必须补齐的字段
- 附录同步动作：本附录已记录可复现步骤、观测结果、推理过程、产出结论和剩余未知点
- 下一步影响：只有当任务 3 更新为已完成时，才允许把关于当前目标的定性判断写入结论；若仍未完成，则只记录剩余检查项
```

## 可复现性说明

- 让另一个 AI 仅靠这个附录就能把当前任务的实验重新跑出来。
- 一个附录只服务一个任务，不要把多个不同任务的验证混在一起。
- 不要只贴 payload 或样本片段；要给出触发它的完整上下文。
- 如果结果依赖环境差异，明确写出环境假设。
- 如果阻塞只是外部连通性异常，不要把它记为任务失败或附录材料。
- 如果实验尚未能直接定性，明确记录剩余未知点，但不要提前把任务标成 `已完成`。
- 未更新主文档时不得继续下一轮验证；必须先补齐主文档同步动作或写明未同步原因。
