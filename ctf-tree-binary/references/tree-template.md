# 主树模板

把它作为 `ctf-worklog/tree.md` 的默认形态。文件名保留 `tree.md` 是为了兼容旧工作流，内容实际维护任务-结论关系图。

## 规则

- 主文档分成 `## 任务` 与 `## 结论` 两部分。
- 任务序号和结论序号分别独立递增，且都从 `1` 开始。
- `任务 1` 固定为环境基线采集任务，`结论 1` 固定为环境基线结论。
- 只有 `已确认` 结论可以作为后续任务的前置结论。
- 每个任务固定写成 `目标`、`前置结论`、`产出结论`、`证据`、`附加` 五行。
- 每个结论固定写成 `内容`、`产出任务`、`证据`、`后续任务` 四行。
- 真正的依赖关系只看 `前置结论`、`产出结论`、`产出任务` 和 `后续任务` 字段，不看序号大小或相邻位置。
- 任务的附录文件名绑定任务序号，例如 `./appendices/task-3.md`。

## 每轮同步检查清单

每次观察、实验、题目相关失败尝试、状态变化、结论变化或任务拆分后，先完成以下检查，再继续下一步：

- 如果阻塞原因只是 Docker Hub、靶机或其他外部基础设施连通性异常，先在对话里告知用户；不要创建任务、结论或附录，也不要进入下面的文档同步流程。

- 主文档已同步：相关任务的状态、证据、产出结论和附加链接已更新。
- 结论块已同步：新增、确认、否定或复核的结论已更新 `产出任务`、`证据` 和 `后续任务`。
- 附录已同步：需要可复现上下文、题目相关失败细节、命令、调试记录、payload、样本或剩余未知点时，已更新对应 `appendices/task-N.md`；外部连通性异常不落档。
- 双向引用已自检：任务的 `产出结论` 对应结论的 `产出任务`，结论的 `后续任务` 对应任务的 `前置结论`。
- 未更新文档不得继续静态分析、动态调试、利用、切换新方向或输出阶段性结论。

## 骨架

```md
# CTF Tree Binary 任务-结论验证链

## 任务

任务 1 [已完成] 目标：完成目标二进制环境的基线采集，确认架构、保护项、附件、交互方式、依赖库与可见约束
前置结论：无；环境基线采集任务
产出结论：结论 1
证据：架构、保护项、附件、远程信息、libc/loader、源码或容器线索等客观环境事实均已记录
附加：[appendix](appendices/task-1.md)

任务 2 [已完成] 目标：确认目标 ELF 的保护项、动态链接关系与关键装载信息是否稳定可复现
前置结论：结论 1 [已确认]
产出结论：结论 2，结论 3
证据：`file`、`checksec`、`ldd` 与运行结果一致，确认了架构和主要保护项
附加：[appendix](appendices/task-2.md)

任务 3 [待执行] 目标：验证可控输入触发崩溃时返回地址偏移是否稳定为同一数值
前置结论：结论 2 [已确认]
产出结论：无
证据：待执行
附加：[appendix](appendices/task-3.md)

任务 4 [已完成] 目标：核验反编译中疑似异或解码函数是否真的负责恢复 flag 校验材料
前置结论：结论 1 [已确认]
产出结论：结论 4
证据：静态交叉引用与动态断点都表明该函数只处理欢迎信息，不参与校验路径
附加：[appendix](appendices/task-4.md)

## 结论

结论 1 [已确认] 内容：目标二进制环境已完成基线采集，确认了架构、附件、交互方式与当前可见依赖线索
产出任务：任务 1 [已完成]
证据：环境基线任务已记录架构、关键交互结果、可用附件、运行依赖与题面约束
后续任务：任务 2，任务 4

结论 2 [已确认] 内容：目标 ELF 未启用 PIE，且主要保护项信息在本地分析结果中稳定一致
产出任务：任务 2 [已完成]
证据：`checksec` 输出与装载地址观察一致，重复运行未出现保护项判定差异
后续任务：任务 3

结论 3 [已确认] 内容：目标程序动态依赖指定的 libc 与 loader 组合，远程复现时需要保持一致
产出任务：任务 2 [已完成]
证据：`ldd` 与题目附件提供的依赖文件对应一致
后续任务：无

结论 4 [已否定] 内容：反编译中疑似异或解码函数负责恢复 flag 校验材料
产出任务：任务 4 [已完成]
证据：断点命中时函数输入只来自欢迎路径，后续未进入校验逻辑
后续任务：无
```

## 编写说明

- 宽泛方向例如“这里可能有栈溢出”或“这段逻辑可能是在做字符串解密”不要直接写进主树；先改写成足够小、可独立执行的任务。
- 已得到的事实写成结论，下一步验证工作写成任务。
- 一个任务产出多个彼此独立的事实时，拆成多个结论。
- 多个任务支撑同一事实时，更新已有结论的 `产出任务` 与 `证据`，不要重复造相同结论。
- 如果任务还没做完，但已有中间材料，把长推理、`gdb` 输出、崩溃现场、寄存器快照、反编译片段或题目相关失败尝试写进附录，不要提前把任务改成 `已完成`。
- 如果失败只是 Docker Hub、靶机或其他外部基础设施连通性异常，直接告知用户，不要写进主树或附录。
