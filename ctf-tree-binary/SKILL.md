# CTF Tree Binary 任务-结论验证链文档

仅在正在解决真实的 CTF Pwn 或 Reverse challenge 且需要维护可续接的任务-结论关系图、appendix/worklog 时使用本 skill；即使用户没有显式调用 `$ctf-tree-binary` 也可以触发。不要用于 web、misc、crypto、forensics、赛后闲聊或泛泛安全讨论。

## 强制同步协议

本 skill 的核心目标是让另一个 AI 能只依赖磁盘上的 `ctf-worklog/tree.md` 和附录继续接手。文档同步不是最终总结，而是每轮推进的硬闸门。

- 启动新题或新调查分支时，先检查 `ctf-worklog/tree.md` 是否存在；不存在时，在完成环境基线采集后立即创建主文档和必要的 `ctf-worklog/appendices/task-1.md`。
- 每次观察到有意义的新信息、完成一次实验、遇到题目相关失败尝试、改变任务或结论状态、拆分后续任务、发现剩余未知点之后，必须先判断并执行文档同步。
- 如果遇到 Docker Hub 拉取/认证/解析/超时/连接重置/限流、靶机或远程服务的 DNS 失败、超时、连接拒绝、路由不可达、握手失败，或其他明显不属于题目逻辑本身的外部基础设施连通性异常，先在对话里明确告知用户当前阻塞点；不要为这类问题创建任务、结论或附录，也不要把它们写入 `ctf-worklog/tree.md`。
- 未更新文档不得继续静态分析、动态调试、写利用脚本、切换新方向、输出阶段性结论或请求后续操作。
- 同步顺序固定为：先更新 `ctf-worklog/tree.md` 的任务-结论关系、状态、证据和双向引用，再创建或刷新对应 `appendices/task-N.md`。
- 如果暂时没有新结论，也要把题目相关失败尝试、中间观察、剩余未知点或收窄后的任务写入当前任务附录；主文档中对应任务的 `证据` 或 `附加` 也要保持同步。
- 每次同步后都检查双向引用：任务的 `产出结论` 必须对应结论的 `产出任务`；结论的 `后续任务` 必须对应任务的 `前置结论`。
- 只有在主文档已同步、必要附录已同步、双向引用已自检后，才允许继续下一轮验证。

## 概述

这个 skill 用来在解决 CTF Pwn 或 Reverse 题时维护一套可持续接手的文档体系。它不负责替你选择利用路线，而是把调查过程整理成 AI 友好的“任务-结论验证链”：任务记录可复现的验证工作，结论记录由任务证据支撑、可被后续任务复用的事实判断。另一个 AI 应能只依赖磁盘上的关系图和附录继续推进。

如果还需要静态分析、调试、脚本验证、payload 设计或利用技巧，请同时使用适合当前题型的解题 skill；本 skill 只负责任务-结论结构、状态流转、证据归档和附录可复现性。

## 与其他解题 skill 协同

- `$ctf-tree-binary` 只负责把调查过程整理成低歧义、可续接的任务-结论关系图，不负责替你决定利用路线。
- 与 `$ctf-binary` 同时使用时，以本 skill 的文档格式和状态规则为准；外部得到的调试结论、IDA 观察、崩溃现场、利用思路、脚本或观察结果，先改写成可独立执行的任务，再把任务产出的可复用事实写成结论。
- 原始终端输出、`gdb` 记录、寄存器/栈/堆快照、反编译片段、脚本、payload 变体、崩溃条件和题目相关失败尝试默认写入对应任务附录；主文档只保留目标、依赖、结论和决定性证据；外部连通性异常先告知用户，不写入 worklog。
- 如果题目核心已经超出 Pwn 或 Reverse 范围，先把当前调查关系图收束到可交接状态，再切换主导 skill。

## 默认工作目录

除非用户明确指定其他位置，默认把工作日志放在：

- `ctf-worklog/tree.md`
- `ctf-worklog/appendices/`

`tree.md` 是兼容旧工作流的文件名；文档内容实际维护任务-结论关系图。

以下参考文件是默认起点：

## 核心模型

- `任务` 是一个足够小、可以独立复现的验证工作，例如“确认目标 ELF 的保护项与动态链接关系”，或“验证输入长度可控时返回地址偏移是否稳定落在 72 字节处”。
- `结论` 是一个由任务证据支撑、可被后续任务复用的事实判断，例如“目标二进制启用了 NX 和 Partial RELRO，但未启用 PIE”，或“返回地址偏移稳定为 72 字节且崩溃点可重复复现”。
- 推理链固定是 `前置结论 -> 任务 -> 产出结论 -> 后续任务`。
- 一个任务可以产出多个结论；一个结论可以由一个任务产出，也可以由多个任务产出、复核或共同支撑。
- 不要用任务表达已经得到的事实；不要用结论表达下一步要做什么。

## 序号约定

- `任务序号` 采用全局递增的自然数 `1`、`2`、`3`，从 `1` 开始，每新增一个任务加 `1`。
- `结论序号` 采用全局递增的自然数 `1`、`2`、`3`，从 `1` 开始，每新增一个结论加 `1`。
- 任务序号和结论序号分别独立递增，永不复用，也不承载层级语义。
- 附录文件名绑定 `任务序号`，例如 `ctf-worklog/appendices/task-3.md`。
- 不得通过序号大小、相邻位置或附录文件名推断依赖关系；真正的依赖只看 `前置结论`、`产出结论`、`产出任务` 和 `后续任务` 字段。

## 启动要求

开始一题新的挑战，或开始一条新的调查分支时：

1. 先按题型收集环境事实：目标架构、位数、ELF/PE/Mach-O 类型、保护项、入口点、二进制附件、源码、符号、libc、loader、容器、远程服务、协议、题面约束、调试条件、IDA 数据库、核心转储，或任何其他硬观察。
2. 完成环境基线后立即创建 `任务 1 [已完成]`，并产出 `结论 1 [已确认]`。这是强制性的起点，因为它记录的是客观环境事实，不是漏洞猜测。
3. 从 `结论 1` 开始，把大而宽的方向改写成足够小、可独立执行的后续任务。
4. 任务执行后，把可复用事实沉淀为结论。任务可以新增结论，也可以复核或修正已有结论。
5. 只有在有可复现证据时，才把任务改成 `已完成` 或 `已失败`，把结论改成 `已确认` 或 `已否定`。在此之前任务保持 `待执行`，结论保持 `待确认`。
6. 如果一个 `待执行` 任务仍然需要继续拆分，说明它写得太宽。应先收窄该任务本身，或把补充验证细节写进附录，而不是提前产出结论。

## 状态语义

任务状态只能使用以下三种：

- `待执行`：任务尚未完成可复现实验，或证据不足以更新任务状态。
- `已完成`：任务目标已经被可靠执行；它可以产出 `已确认` 或 `已否定` 结论。
- `已失败`：实验本身未能可靠执行，例如题目相关验证步骤无法稳定复现、样本损坏、缺少必要依赖或调试条件不满足。它不等于目标被证伪，不得产出关于任务目标的定性结论；但可以产出失败原因类结论。不要用它记录 Docker Hub 连接异常、靶机不可达或其他题目无关的外部基础设施阻塞。

结论状态只能使用以下三种：

- `待确认`：已有候选判断，但还缺少足够证据。
- `已确认`：判断被一个或多个任务的可复现证据支持。
- `已否定`：判断被一个或多个已完成任务的可复现证据否定。

## 外部连通性例外

以下情况默认视为题目无关的外部环境问题，而不是题目调查材料：

- Docker Hub 拉取、认证、解析、超时、连接重置或限流
- 靶机或远程服务的 DNS 失败、超时、连接拒绝、路由不可达或握手失败
- 其他明显不属于题目逻辑本身的外部基础设施连通性异常

处理规则：

- 涉及 Docker Hub 的连接、拉取、认证、解析或 manifest 检查时，优先使用系统 `7897` 代理。
- 第一时间在对话中告知用户当前阻塞点。
- 不要把这类问题写入 `ctf-worklog/tree.md`。
- 不要把这类问题写入 `ctf-worklog/appendices/task-N.md`。
- 不要把这类问题建模成题目任务、题目结论、失败证据或“后续 AI 不应重复”的失败分析。
- 只有在连通性恢复、重新回到题目相关验证后，才继续正常的任务-结论同步流程。
- `worklog` 只记录题目事实、题目验证、题目分支和题目相关失败，不记录与题目无关的外部环境噪音。

## 强制规则

- 主要读者是 AI，不是人。写法要低歧义、字段固定、措辞稳定。
- 主文档是任务-结论二分关系图，不是叙事型 walkthrough，也不是待办清单。
- 每个任务都必须具有一个 `任务序号`；每个结论都必须具有一个 `结论序号`。
- 除 `任务 1` 外，每个任务必须拥有至少一个前置结论，并在 `前置结论` 行中显式引用全部结论序号。
- 除 `结论 1` 外，每个结论必须拥有至少一个产出任务，并在 `产出任务` 行中显式引用全部产出、复核或共同支撑该结论的任务序号。
- 只有 `已确认` 结论可以作为后续任务的前置结论。
- `待确认` 和 `已否定` 结论不得作为后续任务的前置结论。
- `待执行` 任务不得产出已定性的结论。
- 一个任务产出多个互相独立的事实时，必须拆成多个结论。
- 一个结论可以只由一个任务产出；当多个任务产出、复核或共同支撑同一事实时，更新同一个结论的 `产出任务` 和证据，不要重复创建语义相同的结论。
- 任务或结论状态变化时，直接原地更新；不要额外新增一个仅重复裁决的任务或结论。
- 已失败任务和已否定结论要保留在关系图里，防止后续 AI 重复踩坑。
- `已失败` 任务只用于记录题目调查链上的失败实验，不用于记录 Docker Hub、靶机连通性或其他题目无关的外部阻塞。
- 主文档保持精瘦：任务只写 `目标`、`前置结论`、`产出结论`、`证据`、`附加` 五行；结论只写 `内容`、`产出任务`、`证据`、`后续任务` 四行。
- 长推理、终端输出、`gdb` 会话、脚本、样本片段、反编译片段、payload 集合和题目相关失败细节都放任务附录，不放主文档；外部连通性异常不落档。

## 主文档约定

在 Markdown 中维护 `## 任务` 和 `## 结论` 两个列表。序号是稳定身份标识，不是自动编号列表，也不是层级路径。

每个任务必须严格按以下顺序书写：

1. `任务 序号 [状态] 目标：...`
2. `前置结论：...`
3. `产出结论：...`
4. `证据：...`
5. `附加：...`

每个结论必须严格按以下顺序书写：

1. `结论 序号 [状态] 内容：...`
2. `产出任务：...`
3. `证据：...`
4. `后续任务：...`

字段规则：

- 初始任务的 `前置结论` 固定写成 `前置结论：无；环境基线采集任务`。
- 初始结论的 `产出任务` 固定写成 `产出任务：任务 1 [已完成]`。
- 非初始任务的 `前置结论` 必须写成 `前置结论：结论 2 [已确认]` 或 `前置结论：结论 2 [已确认]，结论 5 [已确认]` 这种显式依赖格式。
- 非初始结论的 `产出任务` 必须写成 `产出任务：任务 3 [已完成]` 或 `产出任务：任务 3 [已完成]，任务 7 [已完成]` 这种显式来源格式；单任务产出的结论只写一个任务，多任务产出或复核的结论写全部相关任务。
- `产出结论` 行列出当前任务直接新增、确认、否定或复核的结论序号；没有产出时写 `产出结论：无`。
- `后续任务` 行列出直接使用当前结论作为前置的任务序号；暂无后续时写 `后续任务：无`。
- 如果任务有附录，就在 `附加` 行写指向任务序号文件的 Markdown 链接；没有则写 `附加：无`。

## 附录约定

任务出现以下任一情况时，创建或更新对应附录：

- 完整命令、`gdb` 交互、`pwntools` 脚本、样本片段或代码
- 详细推理过程
- 后续 AI 不应重复的题目相关失败分析
- 需要保存的寄存器、栈、堆、内存布局、崩溃偏移、断点观察或反编译片段

如果阻塞原因只是 Docker Hub、远程靶机或其他外部基础设施连通性异常，不要创建或更新对应附录；先在对话中告知用户。

每个附录只属于一个任务，且文件名必须绑定该任务的 `任务序号`。

每个附录必须按以下顺序包含这些部分：

1. `任务标识`
2. `目标`
3. `前置条件`
4. `可复现步骤`
5. `观测结果`
6. `推理过程`
7. `产出结论`
8. `结果`

可复现性规则：

- `任务标识` 中必须写清：当前任务序号、前置结论列表、当前状态、产出结论列表。
- 初始任务附录的前置结论列表写 `无`。
- 每个附录只服务一个任务；不要把多个不同任务的验证混在同一附录里。
- 尽量给出可直接复制执行的命令、调试脚本、`pwntools` 交互、输入样本和样本处理步骤。
- 写明会实质影响结果的架构、libc、loader、断点位置、输入长度、环境变量、ASLR 状态、题目容器、远程端口和交互顺序。
- 如果结果依赖时间窗口、并发、随机种子或步骤顺序，必须明确指出。
- 如果原始输出太长，只保留决定性片段和足够的复验上下文，不依赖聊天记录补全。
- `产出结论` 部分必须列出本任务新增、确认、否定或复核的每个结论及其状态；如果没有结论，明确写 `无` 并说明原因。
- `结果` 部分必须明确区分三类任务状态：保持 `待执行`、更新为 `已完成`、更新为 `已失败`。
- 只有当当前任务更新为 `已完成` 时，才允许在 `结果` 中声明关于任务目标的定性结论；如果任务更新为 `已失败`，只能记录题目相关失败原因、剩余未知点或下一次验证要补的条件。

## 更新循环

每次获得有意义的观察或完成一次实验后：

1. 如果主文档还不存在，先收集环境信息并创建 `任务 1 [已完成]` 与 `结论 1 [已确认]`。
2. 如果新信息只是 Docker Hub、靶机或其他外部基础设施连通性异常，先告知用户并暂停文档同步；连通性恢复后再继续本循环。
3. 判断新信息是在更新现有任务状态、更新现有结论状态、收窄现有任务，还是基于一个或多个已确认结论导出一个新的后续任务。
4. 如果是新增任务，分配下一个可用的 `任务序号`，并写清它依赖的前置结论。
5. 如果任务产出新事实，分配下一个可用的 `结论序号`，并在任务的 `产出结论` 与结论的 `产出任务` 中互相引用。
6. 如果任务复核或修正已有结论，更新该结论的 `产出任务`、`证据` 和 `后续任务`；单任务结论保持一个产出任务，多任务结论列出全部产出、复核或共同支撑的任务，不要重复创建语义相同的结论。
7. 如果一个任务产出多个互相独立的事实，创建多个结论，不要把多个事实塞进一个结论。
8. 立即更新主文档；这是本轮循环的结束闸门，主文档未同步时不得继续下一轮验证。
9. 只要可复现性、题目相关失败细节、详细推理、命令、调试记录、payload、样本或剩余未知点变重要，就同步创建或刷新任务附录。
10. 保证附录里的 `任务标识`、`产出结论` 和 `结果` 与主文档中的任务状态、前置结论、产出结论一致。
11. 检查双向引用：任务的 `产出结论` 与结论的 `产出任务` 相互对应，结论的 `后续任务` 与任务的 `前置结论` 相互对应。
12. 确认主文档已同步、必要附录已同步、双向引用已自检后，才允许继续静态分析、动态调试、利用或输出阶段性结论。

扩展分支时：

- 第一步扩展通常是从环境基线结论提炼出第一个或少量几个后续任务。
- 在新增任务前，先确认全部前置结论已经是 `已确认`，并在 `前置结论` 行中显式引用其结论序号。
- 同一前置结论下的多个后续任务，必须是彼此独立的验证工作，而不是同一任务的多个操作步骤。
- 多前置结论只在新任务确实需要多个已确认事实共同支撑时使用，不要为了整理方便强行合并前置结论。
- 如果当前任务仍处于 `待执行`，不要提前产出结论；把剩余检查写入附录，或直接收窄这个任务自身。
- 如果任务完成或失败，就原地修改该任务状态，不要新建一个仅仅重复裁决的任务；但外部连通性异常不进入这套任务状态流转。

## 完成

拿到 flag 后：

- 把 worklog 一直保持到终点任务和终点结论仍然自洽
- 直接输出 flag
- 除非用户要求，否则不要强制追加复盘总结

## 需要避免的失败模式

- 跳过最初的环境基线任务和环境基线结论
- 把大而模糊的方向直接写成任务
- 把已得到的事实写成任务，而不是结论
- 把操作步骤写成结论，而不是任务附录
- 把未验证内容写成 `已确认`
- 让 `待执行` 任务提前产出结论
- 把目标被证伪误写成 `已失败` 任务，而不是 `已完成` 任务产出 `已否定` 结论
- 用 `待确认` 或 `已否定` 结论派生后续任务
- 非初始任务没有前置结论，或没有明确写出它依赖的是哪些已确认结论
- 非初始结论没有产出任务，或没有明确写出它由一个还是多个任务产出、复核或共同支撑
- 一个任务产出多个事实时强行合并成一个含混结论
- 多个任务产出或复核同一事实时重复创建语义相同的结论
- 靠序号大小、相邻位置或附录文件名推断依赖关系
- 因为插入或重排分支而重命名已有任务序号或结论序号
- 任务中缺少决定性证据
- 把失败任务或否定结论隐藏掉
- 在附录里只存 payload，不写可运行上下文
- 让关键信息只存在于聊天记录里，而不落到文档中
- 把 Docker Hub 拉取失败、镜像解析失败、远程靶机不可达或其他外部连通性问题写进题目 worklog
- 把题目无关的基础设施异常误记为题目任务失败、题目结论、题目证据或“后续 AI 不应重复”的失败分析
