# Assist Development

在开发过程中持续维护项目内开发文档和 Git 变更记录。目标是让人类能理解和扩充功能设计，也让新的 AI 上下文能快速恢复项目状态并继续开发。

核心原则：**功能分区和规范分区都应尽可能拆成小模块记录，再记录这些小模块如何连接整合成完整项目。**

Git 原则：**初始化项目时建立本地 Git 仓库并连接远程仓库；每次完成一次项目修改后，把本轮相关改动提交到本地 Git 仓库。**

## 文档位置

默认文档根目录：

- `<当前工作目录>/Document`

默认目录下固定使用两个分区：

- `<当前工作目录>/Document/功能分区`
- `<当前工作目录>/Document/规范分区`

目录规则：

1. 如果用户显式指定文档目录，优先使用指定目录作为文档根目录，不强制额外拼接 `Document/`。
2. 如果用户未指定文档目录，使用当前工作目录下的 `Document/` 作为默认文档根目录。
3. 无论使用默认目录还是指定目录，都保持 `功能分区` 与 `规范分区` 两个子目录。
4. 如果文档根目录或两个分区目录不存在，应创建所需目录。
5. 若目录已存在，优先增量更新现有文档，不要无故重写或删除旧内容。

## 模块化记录原则

优先把项目拆成可独立理解的小模块，而不是只写一个庞大的总览文档。

小模块可以是：

- 用户可感知的功能单元，例如首页封面、文章概览、文章详情、登录入口、评论区、搜索、后台编辑器。
- 技术实现单元，例如路由结构、内容集合、状态流、组件边界、接口契约、样式系统、构建部署。
- 横向支撑单元，例如鉴权、配置、错误处理、测试验证、数据迁移、AI 续接上下文。

每个小模块文档应尽量说明：

- 目标：这个模块解决什么问题。
- 边界：它负责什么，不负责什么。
- 输入输出：用户输入、数据输入、页面输出、接口输出或状态变化。
- 依赖关系：依赖哪些模块，被哪些模块使用。
- 当前状态：已完成、未完成、待确认、已知限制。
- 验收方式：如何判断这个模块可用或没有回归。

同时维护整合关系：

- 当项目存在多个模块时，应有一个聚合性文档记录模块如何连接成完整项目。
- 功能分区侧重记录小功能如何串成用户流程。
- 规范分区侧重记录技术模块的调用关系、数据流、文件入口和依赖方向。
- 聚合文档可以命名为 `项目功能总览.md`、`模块整合关系.md`、`项目架构.md` 或沿用项目已有命名。

## 两类文档

### 功能分区

面向人类阅读和扩充，重点记录“功能是什么、为什么这样做、做到什么程度”。

功能分区应优先按小功能拆分，例如：

- `首页封面.md`
- `文章概览.md`
- `文章详情.md`
- `用户登录.md`
- `评论功能.md`

每个功能模块应覆盖：

- 功能目标、用户场景和预期行为。
- 功能边界、输入输出和关键交互。
- UI、接口、服务、数据、配置等层面的实现状态。
- 已完成事项、未完成事项、待确认问题。
- 验收标准、测试场景和已知限制。
- 与其他功能模块的连接方式。

功能分区还应维护功能整合关系：

- 用户从哪里进入这个功能。
- 功能完成后流向哪里。
- 它依赖哪些前置功能。
- 它会影响哪些后续功能。
- 多个功能如何组成完整业务流程。

写法要求：

- 用产品/功能语言优先，不要只堆代码细节。
- 优先更新对应小模块文档；没有合适文档时再新建。
- 跨多个功能的变化，应同时更新相关小模块和整合关系文档。
- 保留可扩充结构，方便人类后续补充。

### 规范分区

面向 AI 续接开发，重点记录“代码在哪里、约定是什么、下一步怎么接”。

规范分区应优先按技术模块拆分，例如：

- `路由结构.md`
- `内容集合.md`
- `首页交互模块.md`
- `样式系统.md`
- `接口约定.md`
- `测试与验证.md`
- `AI续接上下文.md`
- `模块整合关系.md`

每个规范模块应覆盖：

- 模块边界、关键入口文件和相关目录。
- 框架约定、目录约定、命名约定和编码规范。
- 数据流、状态流、接口契约和配置来源。
- 与其他技术模块的依赖关系和调用方向。
- 重要设计决策、取舍原因和兼容性约束。
- 常用开发、测试、构建、验证命令。
- 当前上下文恢复提示：已完成、正在做、下一步、风险点。

规范分区还应维护技术整合关系：

- 页面、组件、服务、接口、数据、配置之间如何连接。
- 用户行为如何触发状态变化、接口调用或页面跳转。
- 数据从哪里产生、在哪里转换、最终在哪里消费。
- 模块之间的依赖方向，避免后续开发误连、倒依赖或重复实现。

写法要求：

- 偏向机器可续接，信息要具体、可定位、可执行。
- 记录关键文件路径、命令、约束和失败经验。
- 优先更新对应技术模块文档；跨模块变化要更新整合关系。
- 避免泛泛总结；优先写下一次 AI 接手开发真正需要知道的内容。

## 工作流程

### 开发前

1. 判断当前任务是否涉及开发、代码修改、调试、重构、架构分析或交付续接。
2. 确定项目根目录，检查是否存在 `.git`：
   - 如果不存在 `.git`，执行 `git init` 初始化本地仓库。
   - 如果已存在 `.git`，沿用现有仓库，不重新初始化。
3. 检查远程仓库连接：
   - 先检查 `git remote get-url origin`。
   - 如果 `origin` 已存在，不覆盖已有远程地址。
   - 如果没有 `origin` 且需要创建远程仓库，必须先询问用户仓库名，不能直接用项目目录名静默创建。
   - 如果 `gh` 可用且 `gh auth status` 成功，使用用户确认的仓库名创建默认 private GitHub 仓库，并连接为 `origin`。
   - 如果 `gh` 不可用、未登录、创建失败或仓库名冲突，询问用户远程仓库 URL，再执行 `git remote add origin <url>`。
4. 确定文档根目录：优先使用用户指定目录，否则使用当前工作目录下的 `Document/`。
5. 检查 `功能分区` 和 `规范分区` 是否存在，缺失时准备创建。
6. 判断本次变更属于：
   - 已有小模块的增量更新。
   - 新增小模块。
   - 多个模块之间的整合关系变化。
7. 检查是否已有相关小模块文档和整合关系文档；优先增量更新，缺失时再新建。

### 开发中

1. 当功能目标、实现方案、模块边界或关键约定发生变化时，同步更新项目内相关文档。
2. 功能行为、验收标准、限制和待办写入对应的功能小模块文档。
3. 代码入口、架构约定、命令、数据流和续接提示写入对应的规范小模块文档。
4. 如果变更影响多个模块的连接方式，同步更新功能整合关系或技术整合关系文档。
5. 对不确定内容明确标注“待确认”，不要伪造已确认结论。

### 交付前

1. 补齐本轮开发涉及的小模块状态：已完成、未完成、验证结果和风险。
2. 确认跨模块依赖、用户流程或数据流是否发生变化；发生变化时更新整合关系文档。
3. 在规范分区留下新上下文可继续开发的恢复信息。
4. 检查 `git status`，只暂存并提交本轮相关改动，不自动纳入无关既有改动。
5. 使用简短、具体的提交信息说明本轮变更，例如 `Add article search flow` 或 `Update routing documentation`。
6. 如果本轮没有可提交的项目修改，明确记录原因，不制造空提交。
7. 在最终回复中简要说明已更新或建议更新的项目内文档路径，并说明本地 Git 提交结果。
8. 如果因为指定目录不可用、未找到合适项目目录、Git 身份未配置、远程信息缺失或任务范围限制未能更新文档或提交 Git，应明确说明原因和建议补充内容。

## 文档命名建议

功能分区：

- 按用户功能或业务模块命名，例如 `首页封面.md`、`文章概览.md`、`文章详情.md`、`登录功能.md`、`评论功能.md`。
- 用 `项目功能总览.md` 或已有同类文档记录多个小功能如何组成完整用户流程。

规范分区：

- 按技术模块或续接主题命名，例如 `路由结构.md`、`内容集合.md`、`组件边界.md`、`样式系统.md`、`接口约定.md`、`测试与验证.md`。
- 用 `模块整合关系.md`、`项目架构.md` 或已有同类文档记录模块连接、数据流和依赖方向。

命名规则：

- 已有命名风格优先于本建议。
- 不为过小、一次性的细节制造碎片文档；只有当内容可独立维护、会被后续开发复用或会影响模块边界时才拆分。
- 如果一个文档开始同时覆盖多个不相关职责，应优先拆分成小模块文档，并在整合文档中记录它们的关系。

## 更新原则

- 模块优先：优先记录小模块，再记录模块整合关系。
- 最小充分：只记录对理解功能或续接开发有价值的信息。
- 增量维护：优先追加、修正和重组相关段落，不重复制造碎片文档。
- 连接明确：涉及多个模块时，写清入口、出口、依赖、数据流和调用方向。
- 事实优先：区分已验证事实、推断和待确认事项。
- 路径明确：涉及代码、命令、接口或配置时写清可定位信息。
- 不替代开发：文档记录服务于开发，不应显著干扰主要任务推进。
- Git 留痕：提交只覆盖本轮相关改动；遇到用户已有未提交改动时，先区分来源，不能擅自覆盖、回退或混入无关变更。
