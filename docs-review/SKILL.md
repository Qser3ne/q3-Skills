# Document Review

用于在文档初稿或完稿已经形成后做全文审校，重点检查内容冲突、重复、章节分区和整体结构，而不是机械润色单句。

## Workflow

1. 先完整阅读全文，再判断问题类型和严重程度，不能只扫描局部段落就下结论。
2. 识别文档的主题、目标读者、标题层级、章节边界、内容归属、前后顺序、过渡关系，以及是否存在冲突描述、重复描述或结构断裂。
3. 默认以审校报告方式输出结果，按以下五类组织：
   - `冲突描述`
   - `重复描述`
   - `分区/章节问题`
   - `整体结构优化建议`
   - `待确认项`
4. “分区是否合理”只检查单篇文档内部的标题层级、章节边界、内容归属、顺序和过渡是否合理，不涉及多文件目录规划或知识库信息架构设计。
5. 默认不直接改写正文。只有当用户明确要求“直接修好”“重写”“给修订稿”“输出最终版”等结果时，先给出简短问题摘要，再产出完整优化稿。
6. 在改稿模式下，可以合并重复内容、消解冲突、重排章节、补齐必要过渡语和结构衔接，但不能凭空补充业务事实、数据、结论或外部信息。
7. 对无法从原文确定、缺少依据或明显需要外部确认的信息，必须保留为 `待确认项`，不能自行编造。

## Review Rules

- 优先判断是否真的存在问题；如果全文未发现明显冲突、重复或分区问题，要明确说明“未发现明显问题”，只给可选的轻量优化建议，不能硬造问题。
- 冲突描述重点检查前后结论不一致、范围定义不一致、术语含义漂移、步骤条件互相打架等问题。
- 重复描述重点检查同一知识点是否被多次近似复述，或是否在不同章节中无必要地重复解释。
- 分区/章节问题重点检查某段内容是否放错章节、标题是否覆盖内容、章节顺序是否妨碍理解、过渡是否缺失。
- 整体结构优化建议应聚焦重组方式，例如合并章节、拆分过长章节、调整先后顺序、补一个总览段或结论段，而不是泛泛而谈。
- 改稿时保持原文语言和语气约定，除非原表达方式本身阻碍清晰度。

## Boundary

- `docs-review` 用于完稿后的质检、问题诊断和结构审校，也可以在用户明确要求时继续产出修订稿。
- `docs-supplement` 仍用于把新增信息并入既有文档、重写相关章节并消除追加式补丁痕迹。
- 如果任务核心是把新内容整合进旧文档，优先使用 `$docs-supplement`；如果任务核心是检查现有全文质量与结构，优先使用 `$docs-review`。

## Output Expectations

- 默认输出审校报告，不默认输出完整修订稿。
- 当用户明确要求改稿时，输出顺序固定为“简短问题摘要 + 完整优化稿”。
- 如果某个判断依赖用户未提供的上下文，先把它写入 `待确认项`，再继续给出基于当前文本可确定的分析或修订结果。
