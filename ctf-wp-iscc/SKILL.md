# ctf-wp-iscc

## 用途

读取当前目录材料，生成一篇新的、可复现的中文 CTF Word 题解。输出文件固定为当前目录下 `./<输出前缀>-<题目名称>.docx`，Word 标题使用用户确认的原始题目名。

## 何时使用

仅在用户显式调用 `$ctf-wp-iscc` 时使用本 skill。不要因为用户泛泛提到“wp”“writeup”“解题报告”就自动套用。

## 输入

- 当前目录中的 WP、脚本、截图、日志、命令记录、请求样例和 `ctf-worklog/` 材料。
- 用户确认的题目名称和当前题目的原始附件范围。
- 环境变量 `SKILL-CTF-WP-ISCC-AUTHOR-LINE`、`SKILL-CTF-WP-ISCC-OUTPUT-PREFIX`、可选的 `SKILL-CTF-WP-ISCC-DOCX-VENV`。

## 输出

- 当前目录下的 `./<输出前缀>-<题目名称>.docx`。
- 正文第一行作者信息、第二行题目类型与题目名称、`解题思路` 和可选 `Exp`。
- 至少三张合规截图、中文字体声明、内容完整性和小于 5MB 的成品文件。

## 执行流程

1. 确认题目名字，并据此确定输出文件名。
2. 收集题目类型、服务地址、附件、运行方式、flag 格式、关键截图、请求/响应、命令输出和脚本行为。
3. 先识别当前题目的原始附件集合；如果用户已经明确指定附件范围，以用户当前给出的附件为最高优先级，不要让目录里的旧材料反向定义题目对象。
4. 如果触发附件歧义条件，先向用户确认附件范围；在确认前不要继续写作，也不要让旧 wp、旧脚本或目录命名替你选题目对象。
5. 对 `misc`、`re`、`mobile`、`pwn` 方向，检查旧 wp、旧脚本、旧记录是否明显针对另一份附件；如果不一致，只提炼思路链和突破口，不继承旧附件上的结论、目标对象或利用对象。
6. 再基于当前附件独立整理思路链：`初始观察 -> 第一假设 -> 验证 -> 新线索 -> 新假设 -> 关键突破 -> 完整复现 -> flag`。
7. 如果存在旧 wp、旧脚本或旧记录，只能用来校验事实、补缺细节和提取思路线索；不得用它们决定章节结构、正文措辞、当前附件归属或最终 `Exp`。
8. 如需写 `Exp`，按脚本重写规则重新实现。
9. 生成 Word 文档。优先使用 `SKILL-CTF-WP-ISCC-DOCX-VENV` 环境变量指定的虚拟环境中的 Python3 和 `python-docx`；若不可用，再检查当前 Python 环境是否已安装 `python-docx`。
10. 成稿后先做一轮审稿，并把附件归属、来源说明和旧附件残留作为优先检查项。
11. 生成后复查 `.docx` 包内图片、字体声明、中文显示、内容归属、风格合规性和文件大小；不满足要求时先修复再交付。

## 约束规则

### 目标

- 读取当前目录材料，生成一篇新的、可复现的中文 CTF Word 题解。
- 开始写作前必须先询问题目名字，不要用目录名、旧 wp 标题或附件名代替。
- 输出文件固定为当前目录下 `./<输出前缀>-<题目名称>.docx`，Word 标题使用用户确认的原始题目名；`<输出前缀>` 从 `SKILL-CTF-WP-ISCC-OUTPUT-PREFIX` 环境变量读取。
- 以选手视角按过程推进顺序写出观察、假设、验证、转向和最终解法。
- 如果目录里已有 wp 或解题脚本，只能作为思路线索、补充线索和适当的事实校验材料，不能照搬内容、结构或代码。
- 对 `misc`、`re`、`mobile`、`pwn` 方向，旧 wp、旧脚本和旧记录默认可能针对别的附件；写作、验证、脚本重写和最终结论都必须重新锚定到用户当前明确给出的原始附件。材料足以解题时，最终 `Exp` 必须根据当前附件执行、提取或利用得到 `flag`，不能直接复用已有解题文档或旧脚本里的 `flag`。
- 成稿正文应当像独立完成的一篇题解，不写旧 wp、旧工作记录、旧脚本、原始笔记、工作流附录或“参考了哪些材料”等来源说明。
- 最终 `.docx` 必须小于 5MB，并在生成后完成截图嵌入、中文字体和内容完整性校验。

### 输入材料与附件归属

默认递归扫描当前目录中的下列材料：

- `*.md`、`writeup*`、`wp*`
- `solve*`、`exp*`、`exploit*`
- `*.py`、`*.sh`、`*.js`、`*.go`、`*.java`
- 截图、录屏导出图、流程图等图片
- `ctf-worklog/` 及其附录
- 其他能证明真实解题过程的日志、命令记录、请求样例和输出片段

如果目标输出文件已存在，默认不要把它当作输入材料，避免自我复写。

附件归属必须谨慎：

- 只有题面、题目压缩包、题目源码包、题目容器文件、题目流量包、题目样本，或用户明确指定为“题目给的文件”的内容，才能写成题目附件、题目提供文件、原始附件或等价表述。
- 当前目录里后来出现的分析脚本、验证脚本、补充截图、工作记录、复现脚本、自动生成文件和 AI/人工整理材料，默认都是分析过程产物，不得写成题目附件。
- 即使分析脚本文件名看起来像题目脚本，例如 `*.py`、`solve*.py`、`script*.py`，也不能仅凭扩展名或目录位置写成题目自带脚本；必须有明确证据证明它来自题目原始附件。
- 如果真正的原始附件只有用户指定的某一个或某几个文件，正文叙述必须严格围绕这些文件表述“附件”。
- 如果目录里存在多个都像原始附件的候选文件，而用户没有明确点名附件范围，必须先向用户确认后再继续；不要自行挑一个，也不要让旧 wp、旧脚本或文件名相似度替你做决定。

对 `misc`、`re`、`mobile`、`pwn` 方向再额外加严：

- 旧 wp、旧脚本、旧工作记录即使看起来相关，也默认可能是在解决别的附件、别的安装包、别的样本、别的二进制、别的流量或别的导出文件。
- 这类旧材料只能提供思路链、排查方向、可能的突破口和适当的事实校验线索，不能直接继承它针对另一份附件得出的定位、结论或利用对象。
- 当前题目的原始附件范围一旦确认，正文里的“附件”“题目提供文件”“原始样本”“题目自带脚本”等表述都只能指向这些文件；如果旧材料里的附件名、包名、二进制名、资源路径、样本名或目录结构与当前附件不一致，一律按“思路可参考、对象已切换”处理。
- 关键判断、关键命令、脚本行为、提取结果和最终 flag 来源都必须回到当前附件重新取证后才能写入正文；不能把旧材料中的判断直接搬成当前附件的既成事实。
- 在能够根据当前附件完成解题的情况下，最终 `Exp` 必须真实运行当前附件并由运行结果得到 `flag`；不得从旧 wp、旧脚本、旧截图、旧记录中直接复制 `flag`，也不得把这些材料中的 `flag` 当作当前附件的验证结果。
- 如果反复检查后仍无法根据已有文档、旧脚本或当前附件真实解出 `flag`，必须先向用户说明卡点并请求确认是否启用“非真实复现降级写法”；只有用户明确确认后，才允许重写脚本并加入大量题目细节内容，最后直接输出已知 `flag`。该降级脚本不要求完全符合题目真实逻辑，但仍应尽量围绕附件名、关键字符串、保护信息、资源路径、二进制符号、流量特征、包名、类名或其他可见题目细节组织，避免写成空泛占位脚本。

可优先使用类似下面的扫描方式：

```bash
rg --files . | rg '(solve|exp|exploit|writeup|payload|script|wp|md|py|sh|js|go|java|png|jpg|jpeg|gif)$'
rg -n '(flag\{|CTF\{|ctf\{|sql|ssti|xss|jwt|pickle|deserialize|offset|ROP|curl|requests|pwntools)' .
```

### 写作流程

1. 确认题目名字，并据此确定输出文件名。
2. 收集题目类型、服务地址、附件、运行方式、flag 格式、关键截图、请求/响应、命令输出和脚本行为。
3. 先识别当前题目的原始附件集合；如果用户已经明确指定附件范围，以用户当前给出的附件为最高优先级，不要让目录里的旧材料反向定义题目对象。
4. 如果触发上面的附件歧义条件，先向用户确认附件范围；在确认前不要继续写作，也不要让旧 wp、旧脚本或目录命名替你选题目对象。
5. 对 `misc`、`re`、`mobile`、`pwn` 方向，检查旧 wp、旧脚本、旧记录是否明显针对另一份附件；如果不一致，只提炼思路链和突破口，不继承旧附件上的结论、目标对象或利用对象。
6. 再基于当前附件独立整理思路链：`初始观察 -> 第一假设 -> 验证 -> 新线索 -> 新假设 -> 关键突破 -> 完整复现 -> flag`。
7. 如果存在旧 wp、旧脚本或旧记录，只能用来校验事实、补缺细节和提取思路线索；不得用它们决定章节结构、正文措辞、当前附件归属或最终 `Exp`。
8. 如需写 `Exp`，按下文脚本重写规则重新实现。
9. 生成 Word 文档。优先使用 `SKILL-CTF-WP-ISCC-DOCX-VENV` 环境变量指定的虚拟环境中的 Python3 和 `python-docx`；若不可用，再检查当前 Python 环境是否已安装 `python-docx`。
10. 成稿后先做一轮审稿，并把附件归属、来源说明和旧附件残留作为优先检查项。逐段确认正文、图片邻近说明与脚本引导语没有把分析过程产物写成题目附件，也没有残留旧附件名、旧包名、旧二进制名、旧样本名、旧资源路径或旧目录结构描述；同时逐条核对是否满足下文“风格与禁令”中的要求。发现不符合之处时，先改写、删减或重排，再继续后续校验。
11. 生成后复查 `.docx` 包内图片、字体声明、中文显示、内容归属、风格合规性和文件大小；不满足要求时先修复再交付。

脚本重写应至少在以下一类结构上有实质变化：函数拆分、请求封装、CLI 入口、重试/超时、日志输出、提取流程或数据组织。除非用户另有要求，不强制单独保存脚本文件。除下方“非真实复现降级写法”外，正文中的完整新脚本必须在当前题目环境或当前附件上实际验证可得到 `flag` 或关键结果。对 `misc`、`re`、`mobile`、`pwn` 方向，如果旧脚本针对的是别的附件，新的 `Exp` 或辅助脚本必须面向当前附件重新设计和验证，不能只替换文件名、路径、包名或样本名后继续沿用旧逻辑。

例外：对 `misc`、`re`、`mobile`、`pwn` 方向，如果已经向用户确认启用“非真实复现降级写法”，则正文中的 `Exp` 可以不符合题目真实逻辑，但必须是重新写出的脚本，包含尽可能多的当前题目细节，并直接输出最终 `flag`。

### 正文格式

正文必须使用中文，并严格按下面顺序组织：

1. 第一行固定写成 `SKILL-CTF-WP-ISCC-AUTHOR-LINE` 环境变量的值。
2. 第二行写成：`题目类型+题目名称`，例如 `Web 题目名`
3. 接下来直接写标题：`解题思路`
4. 解题思路下按 `1.XXX`、`2.XXX`、`3.XXX` 展开，允许每项内用少量小段落分隔关键阶段，但不要再引入新的大标题体系。
5. 如果有利用脚本，最后写标题：`Exp`，直接粘贴完整代码，不允许用截图代替；没有利用脚本时可以不写这一段。

用户给出的括号说明，例如“必须包含文字说明+截图”“如有，请粘贴完整代码，不允许截图！”，只是格式指令，不能写进最终 wp。

### 环境变量

涉及个人身份或输出命名的信息必须从环境变量读取，不要硬编码在 skill 或生成脚本中。

- `SKILL-CTF-WP-ISCC-AUTHOR-LINE`：正文第一行作者信息。
- `SKILL-CTF-WP-ISCC-OUTPUT-PREFIX`：输出 `.docx` 文件名前缀。
- `SKILL-CTF-WP-ISCC-DOCX-VENV`：生成 Word 文档时优先使用的 Python 虚拟环境目录。

因为变量名包含连字符，读取时使用 `printenv`，不要使用 shell 参数展开：

```bash
author_line="$(printenv 'SKILL-CTF-WP-ISCC-AUTHOR-LINE' || true)"
output_prefix="$(printenv 'SKILL-CTF-WP-ISCC-OUTPUT-PREFIX' || true)"
docx_venv="$(printenv 'SKILL-CTF-WP-ISCC-DOCX-VENV' || true)"
```

如果作者行或输出前缀变量未设置，必须先询问用户或要求用户设置环境变量；不要猜测、复用旧文件名或把目录名当作默认身份信息。`SKILL-CTF-WP-ISCC-DOCX-VENV` 未设置时可以回退到当前 Python 环境，但不能猜测本机虚拟环境路径。

正文必须包含：

- 转向理由：是什么现象让你选择当前突破口。
- 判断依据：为什么当前方案成立，不能只写“测试发现”“分析可得”。
- 可复现的关键请求、关键命令或关键交互。
- 至少三张合规截图。
- 如有脚本，包含完整、重新实现且实际验证可用的新脚本；已向用户确认启用“非真实复现降级写法”时，脚本按上文降级 `Exp` 要求执行。

不要保留失败尝试。若材料不足以证明某段关键判断，应写“没有保留完整记录”，不要编造。

### 截图要求

所有方向都必须在正文流程中自然插入至少三张本地截图：

- 截图必须直接嵌入 Word 正文流程，不能是链接、占位、题注编号或悬空图片区块。
- 图片要放在对应的观察、验证或转折文字附近；可以单独占一段，但前后要有自然衔接。
- 不要写“图 1”“图 2”之类标注，不要做题注，不要把多张图机械堆到末尾。
- 图片必须使用内联方式插入，避免浮动、环绕、文本框和文字覆盖图片。
- 如果出现图片与文字重叠，优先调整正文位置、独立图片段落、段前段后空白或分页，不要只靠缩小图片规避。
- 生成后必须检查 `word/media/` 和 `document.xml`，确认至少三张图片实际嵌入 docx 包内。

题型差异：

- Web：必须使用网页截图，优先体现页面入口、关键交互、漏洞触发点、后台界面或拿到 flag 的页面状态。优先复用当前目录已有站点截图；没有可用截图时应主动获取并保存为本地文件。拿不到 Web 截图时必须中止并说明未满足强制要求，不能继续产出不合规 wp。
- Misc、二进制及其他非 Web 方向：必须使用终端风格的本地图片，优先体现实际执行的命令和对应输出。材料不足三张时应补做，例如环境确认、脚本运行中间输出、最终 flag 回显。允许生成终端风格图片，不再强制要求真实终端窗口截图；除已向用户确认启用“非真实复现降级写法”外，图片内容必须来自真实执行结果，不能伪造未发生过的命令或输出，也不能只放纯代码、流程图或只有结果没有命令的半截图。降级写法下可以展示降级脚本的实际运行过程和输出，但正文不得暗示它是真实解题链。
- 二进制题只有在关键证据确实需要展示图形界面分析工具时，才额外插入 IDA 等界面截图；这类图不能替代至少三张终端截图的最低要求。

非 Web 方向如果必须补做终端风格图片，视觉上应尽量贴近 Ubuntu 默认终端，只保留终端内容区域、提示符、命令与输出；不要加入“基线确认”“关键步骤”“最终结果”等标题类文字、阶段名、说明标签或额外装饰。

### Word 与中文字体校验

生成 `.docx` 后必须主动检查正文中文、代码块中文、截图中文是否存在乱码、方块字、缺字或替换字体异常，不能留给用户手动处理。

默认处理策略：

- Word 正文与标题显式指定稳定中文字体族，例如 `Microsoft YaHei`、`微软雅黑`、`Noto Sans CJK` 或当前环境中可确认存在的中文字体。
- 代码块同时指定西文字体和东亚字体，避免中文注释或中文输出显示异常。
- 生成截图或代码图时，优先使用本机已安装且覆盖中文的字体，不要盲目使用只适合西文的 `DejaVu`。

至少完成一次验证：

- 检查 docx XML 中的字体声明是否符合预期。
- 检查生成图片所用字体是否支持中文。
- 把附件归属和来源说明作为优先检查项，逐段检查成稿正文、截图前后说明和 `Exp` 前后的过渡文字，确保没有把分析过程产物写成题目附件，也没有残留旧附件名、旧包名、旧二进制名、旧样本名、旧资源路径或旧目录结构描述，并逐条核对是否满足“风格与禁令”。
- 如条件允许，重新打开或重新解析成品，确认中文内容没有明显乱码。
- 检查最终 `.docx` 文件大小；超过 5MB 时压缩图片、减少冗余内容或重新排版后再输出。

- 如果作者行或输出前缀变量未设置，必须先询问用户或要求用户设置环境变量；不要猜测、复用旧文件名或把目录名当作默认身份信息。
- 如果目录里存在多个都像原始附件的候选文件，而用户没有明确点名附件范围，必须先向用户确认后再继续。
- 如果反复检查后仍无法根据已有文档、旧脚本或当前附件真实解出 `flag`，必须先向用户说明卡点并请求确认是否启用“非真实复现降级写法”。
- Web 题拿不到合规网页截图时必须中止并说明未满足强制要求，不能继续产出不合规 wp。

## 示例

User:

```text
$ctf-wp-iscc
```

Assistant:

```text
先询问题目名字并确认原始附件范围，读取环境变量，整理当前目录材料，生成合规 Word 题解并完成截图、字体、内容和文件大小校验。
```

## 不适用场景

- 不用于普通 Markdown WP；普通 Markdown WP 使用 `$ctf-wp`。
- 不在用户泛泛提到“wp”“writeup”“解题报告”时自动套用。
- 不用于没有显式调用 `$ctf-wp-iscc` 的场景。

以下风格与禁令仍需遵守：

- 允许详细，但不要故作抒情；保持技术判断清晰，少写空话。
- 优先用可复制的文本、请求、命令和脚本表达，不要过度依赖截图。
- 对关键证据和关键结论之间的因果关系写明白。
- 不要把 Word 文档写成 Markdown 语气或 Markdown 排版说明。
- 不要提及 wp 创建时间、写作时间、复现时间、耗时、日期或时间戳，也不要写“本文写于”“复现于”“当天/当时”等时间导向表述。
- 不要因为已有 wp 存在就直接在它基础上局部修补，不要沿用旧 wp 的章节骨架后只做同义改写。
- 不要复制旧 wp 的大段措辞、过渡句或总结句，不要把旧 wp 当作主材料倒推正文。
- 不要直接复制旧脚本，也不要只改变量名、函数名或注释来伪装“重写”。
- 不要在成稿中提到用户提供但不属于题目原始附件的脚本和文档。
- 不要偏离第一行、第二行、`解题思路`、`Exp` 的顺序和措辞。
- 不要输出 `.md`，也不要要求后续再手工转换格式。
- 不要伪造未发生过的操作或结论，不要只给 payload 而不解释为什么是它；仅在 `misc`、`re`、`mobile`、`pwn` 方向已获得用户明确确认启用“非真实复现降级写法”时，才允许使用上文规定的降级 `Exp` 直接输出 `flag`。
