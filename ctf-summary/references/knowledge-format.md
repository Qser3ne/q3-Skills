# CTF 知识点文章格式

使用本参考文件检查 `$ctf-summary` 的最终 Markdown。该格式用于从一道 CTF 题目的 WP 中沉淀一个可复用知识点，不用于复写完整解题报告。

## 固定结构

```markdown
# <知识点标题>

## 知识点概述

用一两段说明这个知识点的核心模型、利用价值和常见形态。这里应抽象成可迁移经验，而不是直接讲原题步骤。

## 来源场景

说明知识点来自哪道题、哪类题或哪个 WP 片段。保留能支撑知识点的关键现象、代码、配置、请求、payload 或输出。

## 适用场景

列出该技巧或漏洞模型成立所需的条件，例如可控输入、可读字段、执行上下文、过滤规则、版本差异、权限状态或业务前置状态。

## 最小复现场景

给出能解释知识点的最小本地示例、伪服务、代码片段或交互模型。写入前应验证该场景确实能支撑知识点；能容器化验证时，优先用本机 Docker 构建隔离容器测试。不要默认把 Docker 构建过程写成正文重点；只有用户明确要求时才补充完整 Docker 复现。

## 利用步骤

按复现顺序写出关键操作。每一步说明动作、原因和预期效果，避免只堆命令。

## 完整 POC

给出可复制的关键请求、脚本或命令。若缺少真实环境、依赖或运行条件，必须显式说明。

## 复现结果

给出验证证据、真实回显、已有 WP 结果或预期结果。能实际执行时必须执行后再写结果；能用 Docker 容器验证时，优先写容器内测试得到的关键回显。未实际执行的内容必须标注“预期”或“未动态验证”，不能写成确定结论。

## 利用边界与注意点

总结该知识点的限制条件、不适用场景、常见误判、失败原因和防护边界。
```

## 写作约束

- 默认产出路径为 `/mnt/d/知识库/CTF_Web/Web_CTF/临时/ctf-summary-<知识点>.md`。
- 标题保持固定，不删除、不改名。
- 每篇文章优先沉淀一个核心知识点；如果 WP 中存在多个独立知识点，拆成多篇或先询问用户。
- “来源场景”只保留必要证据，不按时间线写完整 WP。
- “最小复现场景”追求短小、可解释、可迁移，不照搬整题环境。
- 写入文档的知识点必须经过验证，至少能追溯到动态测试回显、Docker 容器内测试结果、源码分析依据、WP 原始证据或明确的不确定说明。
- 命令输出和复现结果不能编造；没有真实执行时明确写“预期结果”或“未动态验证”。
- 边界部分必须写清楚成立条件和失败条件。

## 输出检查清单

写完文档之后，必须按下面清单进行自检：

- [ ] 是否包含全部要求标题：知识点概述、来源场景、适用场景、最小复现场景、利用步骤、完整 POC、复现结果、利用边界与注意点
- [ ] 是否进行了可行的动态验证；可容器化时是否进行了 Docker 容器内验证
- [ ] 如果未进行 Docker 容器内验证，是否说明原因，例如 Docker 不可用、镜像无法拉取、依赖无法安装或该知识点只能静态验证
- [ ] 复现结果是否来自真实执行、容器内测试、已有 WP 证据，或已明确标注为预期 / 未动态验证
- [ ] 完整 POC 是否可复制，缺少环境、依赖或运行条件时是否已说明
- [ ] 适用场景是否写清楚成立条件
- [ ] 利用边界与注意点是否写清楚失败条件、限制和防护边界
- [ ] 是否没有编造版本条件、命令回显或复现结果
