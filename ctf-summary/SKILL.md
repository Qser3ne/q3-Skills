# CTF 知识点总结

仅在用户显式调用 `$ctf-summary` 时使用本 skill。不要因为用户泛泛提到“CTF”“writeup”“题解”“知识总结”就自动套用。

## 目标

- 从一道 CTF 题目的 WP、题解、复现记录、脚本、payload、请求样例、日志或命令输出中提炼可复用知识点。
- 输出中文 Markdown 知识点文档，而不是重新写一篇完整 WP。
- 按参考格式生成轻量知识库文章，突出知识点概述、来源场景、适用场景、最小复现场景、利用步骤、完整 POC、复现结果、边界与注意点。
- 写入文档的知识点必须先经过验证，确认原理、条件、payload 或复现逻辑无误。
- 能够本地复现时，优先通过本机 Docker 构建隔离容器并在容器中测试；只有用户明确要求完整 Docker 复现文档时，才把 Docker 构建过程作为正文重点展开。

## 默认产物

默认输出到 `/mnt/d/知识库/CTF_Web/Web_CTF/临时/ctf-summary-<知识点>.md`。

- 如果无法从材料中确定知识点标题，先询问用户文件名或知识点主题。
- 如果目标文件已存在，先询问用户覆盖、改名还是追加，不要擅自覆盖。
- 如果用户在具体任务中明确指定其他输出路径，以用户当次指令为准；否则使用该固定目录。

## 输入材料收集

优先扫描当前目录中的材料：

```bash
rg --files . | rg '(writeup|wp|solve|exp|exploit|payload|script|Dockerfile|docker-compose|md|txt|log|py|sh|js|java|go|php|rb)$'
rg -n '(flag\{|CTF\{|ctf\{|sql|ssti|xss|csrf|ssrf|rce|deserialize|pickle|jwt|upload|bypass|sandbox|xxe|curl|payload|exploit|poc)' .
```

需要提取的信息：

- 题目名、比赛名、题型、附件或服务形态。
- WP 中真正可迁移的漏洞、技巧、利用模型或排查方法。
- 关键触发点、payload、请求样例、命令样例、脚本片段和结果输出。
- 知识点成立所依赖的版本、配置、权限、输入格式、过滤规则、业务状态或可控字段。
- 能够支撑最小复现的服务行为、代码片段、协议交互或本地模拟方式。

如果材料不足以确定漏洞原理、成立条件、关键 payload 或复现方式，先说明缺失项并请求用户补充，不要编造。

## 知识点验证要求

写入最终 Markdown 前必须验证知识点是否成立。验证优先级：

1. 运行材料中已有的复现脚本、测试命令、PoC 或最小服务，确认关键现象和输出。
2. 如果原题环境不可复用，构造本地最小复现场景验证核心机制。
3. 如果本地复现需要依赖服务、特定运行时或危险输入，优先用本机 Docker 构建隔离容器，在容器中运行服务、脚本或验证命令。
4. 如果 Docker 不可用、镜像无法拉取、依赖无法安装或复现受限，记录实际尝试过的命令、失败原因和仍未验证的结论，不要把未验证内容写成确定结论。

验证时遵守：

- 只验证授权的本地材料或用户提供的题目环境，不连接、扫描或攻击真实外部目标。
- 优先验证知识点本身，而不是完整复刻整道题。
- 对能够实际运行的 payload、脚本或最小服务，必须执行后再写“复现结果”。
- 对只能静态验证的知识点，必须说明依据来自源码、WP 证据、协议行为或文档，并标注无法动态复现的原因。
- 最终文档中涉及“可利用”“可绕过”“可执行”“可读取”等确定性表述时，必须有动态测试、静态证据或原 WP 证据支撑。

## 固定文档结构

生成正文前先读取 `references/knowledge-format.md`，按其中的固定标题和写法检查最终文档。

正文必须使用 Markdown，并包含下面这些标题，标题文字必须保持一致：

```markdown
# <知识点标题>

## 知识点概述

## 来源场景

## 适用场景

## 最小复现场景

## 利用步骤

## 完整 POC

## 复现结果

## 利用边界与注意点
```

可以在这些章节下增加必要的三级标题，但不要删除或改名固定标题。

## 写作流程

1. 确认用户明确调用了 `$ctf-summary`。
2. 扫描当前目录材料，识别原始 WP、复现记录、脚本、payload、关键请求和日志。
3. 提炼单一核心知识点：漏洞类型、核心原理、触发入口、利用约束、成功标志和防护边界。
4. 将原题细节压缩进“来源场景”，只保留能解释知识点来源的证据，不按完整时间线复写 WP。
5. 验证知识点：能运行就实际运行；能容器化就优先用本机 Docker 隔离测试；无法动态验证时写清楚静态证据和限制。
6. 设计“最小复现场景”：优先使用本地安全示例、伪服务、最小代码或等价交互说明；不要求把 Docker 构建过程写入正文，除非用户明确要求。
7. 写出利用步骤、完整 POC 和复现结果；已有真实命令回显时保留真实回显，没有实际执行时明确写“预期结果”或“未实际执行”，且不能把预期当作已验证结论。
8. 写出利用边界与注意点，说明成立条件、不适用场景、容易误判处和防护思路。
9. 写入目标 Markdown 前检查 `/mnt/d/知识库/CTF_Web/Web_CTF/临时/ctf-summary-<知识点>.md` 是否已存在。
10. 写完目标 Markdown 后，按固定输出检查清单逐项自检；未满足项必须修正文档，无法满足项必须在最终回复中说明原因。

## 质量要求

- 必须是中文。
- 必须像知识沉淀文档，不要写成完整比赛 WP。
- “知识点概述”要解释原理和通用模型，不要只贴 payload。
- “来源场景”要说明知识点来自哪道题或哪类题，保留必要证据但不要复写完整解题过程。
- “适用场景”要列出可迁移的成立条件。
- “最小复现场景”要足够小，避免把整道题环境原样搬进去。
- “完整 POC”要尽量可复制；如果缺少环境或依赖，明确说明。
- “复现结果”优先使用实际执行结果；如果能用 Docker 容器验证，应在容器中测试后再写入结论。
- “复现结果”只能使用材料中已有结果、实际执行结果或明确标注的预期结果；如果只是推导，必须标注为预期。
- 对每个写入文档的核心知识点，都要能追溯到验证证据：动态测试回显、容器内测试结果、源码分析依据、WP 原始证据或明确的不确定说明。
- “利用边界与注意点”必须抽象出限制条件、失败原因和防护边界。
- 对不确定的版本、条件或结论，明确写“不确定”或请求用户补充，不要伪造。

## 输出自检

写完文档之后，必须按下面清单进行自检：

- [ ] 是否包含全部要求标题：知识点概述、来源场景、适用场景、最小复现场景、利用步骤、完整 POC、复现结果、利用边界与注意点
- [ ] 是否进行了可行的动态验证；可容器化时是否进行了 Docker 容器内验证
- [ ] 如果未进行 Docker 容器内验证，是否说明原因，例如 Docker 不可用、镜像无法拉取、依赖无法安装或该知识点只能静态验证
- [ ] 复现结果是否来自真实执行、容器内测试、已有 WP 证据，或已明确标注为预期 / 未动态验证
- [ ] 完整 POC 是否可复制，缺少环境、依赖或运行条件时是否已说明
- [ ] 适用场景是否写清楚成立条件
- [ ] 利用边界与注意点是否写清楚失败条件、限制和防护边界
- [ ] 是否没有编造版本条件、命令回显或复现结果

## 禁止事项

- 不要把题解按时间顺序完整复写成另一篇 WP。
- 不要强制加入 Docker 最小复现，除非用户明确要求。
- 不要跳过验证直接写入知识点结论。
- 不要在未实际执行的情况下伪造命令回显或复现结果。
- 不要把 Docker 未验证、脚本未运行或仅凭猜测得到的内容写成确定结论。
- 不要编造题目名、payload、漏洞条件、版本条件或 flag。
- 不要只给 payload，不解释为什么它能代表该知识点。
- 不要省略固定标题。
